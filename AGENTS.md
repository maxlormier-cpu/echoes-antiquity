# Codex Project Rules for Echoes of Antiquity

This repository powers **Echoes of Antiquity**, an English-language static history site focused on ancient history, memory, power, myth, sources, and forgotten figures.

These rules are permanent project instructions for future Codex interventions.

## Core Principles

- Preserve the existing historical text unless a change is required for structure, metadata, accessibility, or formatting.
- Never delete article content unless the user explicitly asks for deletion.
- Never invent historical facts, dates, citations, sources, links, quotations, images, or scholarly debates.
- Preserve the site's literary, immersive, historical tone.
- Prefer minimal, coherent, repeatable edits over broad rewrites.
- Keep the site compatible with modern Google Search, AI Overviews, and AI-mode style answers without making the writing mechanical.
- Before broad changes to existing articles, propose a plan and apply it to one test article only. Wait for user validation before generalizing.

## Repository Structure

- Static generator: `tools/build.mjs`.
- Article source files: `content/articles/*.md`.
- Author profiles: `content/authors.json`.
- Site metadata: `content/site.json`.
- Public assets: `public/`.
- Article writing and cleanup checklist: `ARTICLE_GUIDELINES.md`.
- Generated output: `dist/` and should not be edited manually.
- Article display, metadata, sitemap, RSS, canonical URLs, and page head output are generated from `tools/build.mjs`.

## Article Structure Standard

Existing and future articles should be able to support this structure:

1. SEO title.
2. Meta description in the document head.
3. Article H1.
4. Short subtitle or lead when available.
5. `Key answer`.
6. `Why it matters`.
7. Optional `At a glance` table for complex articles.
8. Table of contents generated from H2 headings through the `[[toc]]` marker.
9. Main narrative article body.
10. Optional `Historical debate` section when the topic involves fragile sources, propaganda, legend, debated identity, uncertain dating, or politically oriented inscriptions.
11. `What the sources say`.
12. `Further reading on Echoes of Antiquity`.
13. Sources or references.
14. Author biography generated by the article template.
15. Article or BlogPosting JSON-LD when supported by the generator.

For existing articles:

- Add structure without removing or rewriting the main narrative body.
- Generate `Key answer` and `Why it matters` only from information already present in the article.
- If the article does not contain enough reliable information for a block, add a clean TODO instead of inventing content.
- Do not invent sources or internal links.
- Do not invent images.
- Do not alter historical claims except to fix obvious formatting issues.

For future articles:

- Use frontmatter and article sections consistently.
- Keep metadata factual and specific.
- Keep image alt text descriptive.
- Keep source lists explicit.

## SEO and Metadata Rules

Every article page should provide, either through frontmatter or the generator:

- unique `<title>`;
- meta description;
- canonical URL;
- Open Graph title;
- Open Graph description;
- Open Graph image when an article image exists;
- Twitter Card metadata;
- publication date;
- modified date when available;
- author name;
- tags or categories when supported.

Never place raw meta tags inside the article body.

## Structured Data

When implemented in the generator, Article or BlogPosting JSON-LD should include:

- `@context`;
- `@type`;
- `headline`;
- `description`;
- `image` when available;
- `author`;
- `publisher`;
- `datePublished`;
- `dateModified` when available;
- `mainEntityOfPage`;
- `keywords` when available.

Never generate invalid JSON-LD.

## Reusable Content Blocks

### Key Answer

- Visible heading: `Key answer`.
- 3 to 5 sentences maximum.
- Answer the article's main historical question directly.
- Include who, what, when, where, and why it matters when the article supports it.
- Do not add unsupported claims.

### Why It Matters

- Visible heading: `Why it matters`.
- 2 to 4 sentences maximum.
- Explain historical significance.
- Relate the topic to power, memory, sources, empire, religion, culture, legitimacy, war, or legacy when relevant.
- Avoid generic filler.

### At a Glance

Optional for complex articles. Suggested fields:

- Period;
- Place;
- Main figures;
- Main source;
- Historical issue;
- Legacy.

Only display when reliable fields exist.

Use a Markdown table when the generator supports it.

### Historical Debate

Use only when the article itself contains material on:

- ancient propaganda;
- legendary traditions;
- debated identities;
- fragile chronicles;
- mythological traditions;
- uncertain dates;
- politically oriented inscriptions;
- late or hostile sources.

Do not invent a debate.

### What the Sources Say

When possible, distinguish:

- Primary sources;
- Modern references;
- Historical caution.

Do not create full bibliographic references unless the source is already listed or clearly identifiable.

### Further Reading

- Visible heading: `Further reading on Echoes of Antiquity`.
- Use only real internal articles already present in `content/articles`.
- Use descriptive link text.
- Prefer 3 to 5 links when genuinely relevant.
- Do not show this section if no relevant internal article exists.

### Author Bio

Default biography:

> Maximilien Lormier is a French history teacher, writer, and creator of historical content. His work focuses on ancient history, political power, forgotten figures, and the way historical memory is shaped by sources.

Do not duplicate the biography in each article. Use the reusable template or author profile system.

Author biographies are centralized in `content/authors.json` and rendered by the article template.

## Image Rules

- Every article image should have meaningful alt text.
- Use useful titles or captions when supported.
- Keep images responsive.
- Do not replace existing images unless requested.
- Do not invent images.
- Never present AI-generated illustrations as historical proof or documentary evidence.
- If AI-generated illustrations are used, keep or add a clear editorial note in the site's editorial policy.
- Watch for iCloud placeholder issues: local files may appear as deleted and hidden `.icloud` placeholders. Restore tracked images before committing.

## Trust Pages

The site should include:

- `About Echoes of Antiquity`;
- `Editorial Method`;
- `Sources Policy`.

These pages should explain authorship, educational purpose, source handling, uncertainty, legendary material, and the illustrative status of AI-generated images.

## Technical Quality Checks

Before finishing any code or content change:

- Run `npm run build`.
- Check `git status --short`.
- Ensure no unrelated images are accidentally deleted.
- Ensure no `.icloud` placeholder files are committed.
- Ensure article links and image paths are valid for the generator.
- Ensure the generated article has one H1 and a working canonical URL.
- Do not edit `dist/` manually.

## Publication Rules

- Use the GitHub Desktop bundled Git binary when system Git is blocked by missing Xcode command line tools.
- Push only after the build passes.
- Do not rewrite remote history.
- If files were uploaded at repository root by mistake, move article files into `content/articles/` and assets into `public/assets/articles/`.
